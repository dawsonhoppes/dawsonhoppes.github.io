    • Munchies

    College students typically go out to eat in groups which can be difficult when everyone has different preferences. They also discover restaurants in unconventional ways. Current restaurant applications like Yelp have gotten too big and cloud the discovery experience.
    The Problem


    Indecisive users need a simple way to find the perfect restaurant.
    Context

    This group project was completed for a Programming for the Information Professional course at the University of Maryland. Our team built the app from the ground up, starting with HTML/CSS and adding in JavaScript features.




    Team
    • Dawson Hoppes
    • Zora Moore
    • Simone Pimento
    Challenges

    In order to complete this project we had to connect our app to the Zomato API. No one in our group had previous experience developing with APIs, but we knew to effectively produce our application we would need to incorporate this functionality.

    User Interviews

    Before we started developing our app or creating any solutions we wanted to spend some time with other college students, and better understand their specific needs. We interviewed 5 students and interpreted the data we collected from those conversations.

    Here are some of the most poignant quotes and feelings we captured through our research.

    "I want an app that is quick and easy to use when I'm looking for my restaurant."
    "Apps like Yelp have too many features for me. I am not trying to find a contractor, I just want to find somewhere to eat.
    "I typically don't even know what to search for, I'm too indecisive and just want something that will tell me where to go."

    Solution Brainstorming

    Based off our user interviews, we identified these potential problems with current restaurant applications.

    • Discovery Experience is too clouded.
    • No Interaction with Friends.
    • Collaboration - Allow users to work with friends to compile lists of their favorite restaurants.
    • Combine restaurant discovery and reservation process.
    Initial Proposal

    Due to the timeline of our course, we decided to limit the scope of our application to simplifying the restaurant recommendation experience.


    Project Plan

    We developed a project plan that contained software resources, our project timeline, and our risk management plan to ensure successful and timely execution of our project.

    While some of us had more experience with coding, in order to continue growing our skills we decided to collectively work on all aspects of the application.
    Design




    Approach

    We aimed to design a clean user interface with a minimal aesthetic, that mirrored our goal of a simple experience.


    Sketching & Wireframing

    Results

    Code


    HTML/CSS

    Before we started adding any functionality to the app, we set up our framework and nailed down our style guide and CSS. We went through a few iterations of html until we were finally happy with the result.

    We were able to then use this prototype to conduct short user studies to find out what features we would need to adjust in the future.

    JavaScript

    After we finished working on the structor of our app it was time for us to start working on adding in an API to bring our app to life.

    We spent a few days researching how APIs work, and exactly how we could use the Zomato API to fit our needs. What is great about the Zomato API is that it gives you access to seemingling unlimited amounts of restuarant information, including restaurant websites, menu links, location, price, and more. The hard part of our task was figuring out exactly what information we needed to use.



    API

    Our team decided it would make the most sense to work with the Zomato API. It is a free service that allows you to collect information about thousands of restaurants all over the world. It is able to supply you with a thumbnail for the restaurant, location, menu, website, cost, and so much more.

    When we started adding in the API feature to our application, it became evident that we might need to adjust our original design for the product. Once we started working with the API we realized that we wouldn't be able to link the API data to our circle menus so we had to rethink our main page.

    The design we landed on was a simple and easy to use drop down menu where the user can select their food category, and then enter a city name. After the user clicks the search button, a scrollable list of restaurants appear.
    Final Product


    While we had to adjust our design from our original idea, the final product is a fun easy to use prototype of the munchies application.




    Conclusion

    Working on Munchies taught me how to translate my ideas into working prototypes. It is fun and challenging task, but after working on this project I feel more confident in the language, processes, and structure of code. More specifically this project has taught me:

    • How to develop prototypes with HTML/CSS
    • How to add in JavaScript functionality
    • Developing with an API
    • Knowing when to alter or shift design focus
    • Working with a team and collaborating on code
